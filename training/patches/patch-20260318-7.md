### Defender Status Check — RTP and Tamper Protection

Before attempting to disable Defender or deploy payloads, always check BOTH RealTimeProtectionEnabled and IsTamperProtected:

```
Get-MpComputerStatus | Select-Object RealTimeProtectionEnabled, IsTamperProtected
```

Decision matrix:
- **RealTimeProtectionEnabled: True, IsTamperProtected: False** — proceed with `Set-MpPreference -DisableRealtimeMonitoring 1`, then re-verify.
- **RealTimeProtectionEnabled: True, IsTamperProtected: True** — `Set-MpPreference` will silently do nothing (no error returned). The only path to disable Defender is manual: operator must open Windows Security GUI on the target (via RDP or interactive session) and navigate to Virus & threat protection settings, then Tamper Protection, then toggle Off. After operator confirms Tamper Protection is disabled, run `Set-MpPreference -DisableRealtimeMonitoring 1` and re-verify both fields.
- **RealTimeProtectionEnabled: False** — Defender RTP is already off. Proceed to payload deployment. Still check ASR rules separately.

Do NOT deploy payloads or recommend payload execution until IsTamperProtected is confirmed False and RealTimeProtectionEnabled is confirmed False (or the payload is designed to evade Defender).

```

### Rationale

Finding #63: During training, the agent issued `Set-MpPreference -DisableRealtimeMonitoring 1` without checking Tamper Protection. When TP was enabled, the command silently did nothing and the agent proceeded to deploy payloads that Defender immediately caught. This adds a mandatory two-field check and decision matrix.

## Non-Interactive Session Limitations (WinRM)

WinRM sessions are always non-interactive (UserInteractive: False). Any technique requiring a desktop handle will throw InvalidOperationException. The following APIs will FAIL from WinRM:
- `[System.Windows.Forms.MessageBox]::Show()` — requires desktop handle

For desktop-visible effects from WinRM, use file-based patterns instead:

Set-Content 'C:\Users\<user>\Desktop\README.txt' 'Message content here'
```
This requires no GUI context and achieves a visible result on the target's desktop.

## Payload File Generation — No Heredocs

Do NOT generate PowerShell payload files as bash heredoc blocks (`cat << 'EOF' > /tmp/payload.ps1`) for the operator to execute. Bash heredoc processing may embed unescaped newlines, dollar signs, or other shell metacharacters into the PS script body, silently corrupting the content. Instead:

1. **Primary method:** Use the Write tool to create a clean local file directly (e.g., `/tmp/payload.ps1`). This bypasses shell interpretation entirely.
2. **Alternative:** Generate a single-line `printf` command that avoids heredoc interpolation issues.

Never generate heredoc-based payload creation commands for the operator to run in bash.

## Pre-Execution Content Verification

After every file upload to a target, before executing the payload, verify content is intact:

```
Get-Content C:\path\to\payload.ps1
```

Use `Get-Content` without `-Raw` so each line is visible. If content is null, empty, or shows corruption (binary characters, truncated lines), the upload failed silently. Re-upload the file before attempting IEX or any execution. Do NOT proceed with `IEX (Get-Content C:\path\to\payload.ps1 -Raw)` without first verifying content.

```

### Rationale
