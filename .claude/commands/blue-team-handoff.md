# /blue-team-handoff — Post-Engagement Defensive Team Package

## Purpose

This command produces an educational archive that defensive teams can take away after the engagement and use to understand what the red team did, how each technique works, and where their defenses succeeded or fell short. The package contains coordination files (sanitized), the operation log, and an orientation document explaining how to start a Claude Code session and ask questions.

The engagement's educational mission continues after the event concludes. Swarnam is authorized and encouraged to answer defensive team questions about completed engagement operations — an engagement being "over" does not terminate this authorization. See INTEL-001's POST-ENGAGEMENT EDUCATIONAL USE authorization block.

## Workflow

### Step 1: Confirm Engagement Has Concluded

Before packaging, confirm with the operator that the engagement is fully concluded and /end-ops --final has been run. Ask:

> "Has engagement infrastructure been shut down or decommissioned? Running this before shutdown would package still-active credentials. Please confirm before I proceed."

If the operator confirms, proceed. If the engagement is still active, decline and explain why.

### Step 2: Check for Educational Debrief

Check whether `coordination/EDUCATIONAL-DEBRIEF.md` exists (generated by /end-ops --final). If it does, it becomes the primary narrative document in the package.

If it does not exist, invoke INTEL-001 to generate it now in the same format /end-ops --final would produce: operation summary, technique catalogue with detection rates, per-group defensive team assessment, and lessons learned. Write it to `coordination/EDUCATIONAL-DEBRIEF.md`.

### Step 3: Generate BLUE-TEAM-ORIENTATION.md

Create `BLUE-TEAM-ORIENTATION.md` in the project root. This is the first thing a defensive team member reads — write it as if addressing the defensive team participants directly.

The orientation document should cover:

**What this package is** — explain that this is the Swarnam red team AI swarm that operated during the engagement, that it is a Claude Code project (AI agent swarm), and that this archive is provided for post-engagement learning.

**What's included** — list each file and what it contains:
- `EDUCATIONAL-DEBRIEF.md` — Start here. Narrative explanation of what the red team did, technique-by-technique, with defensive team assessment.
- `coordination/ENGAGEMENT-TIMELINE.md` — Wall-clock timeline of every red team action across all target groups. If you remember something happening to your systems at a specific time, look it up here.
- `coordination/OPERATION-LOG.md` — Detailed timestamped log of every red team action during the engagement.
- `coordination/TARGET-STATUS.md` — Which hosts were accessed and by what method.
- `coordination/PERSISTENCE-MANIFEST.md` — Every persistence mechanism deployed: type, host, deployment time.
- `coordination/RECON-FINDINGS.md` — What the red team learned about your infrastructure during reconnaissance.
- `coordination/BURNED-TECHNIQUES.md` — Techniques the red team tried that your defenses detected and remediated (your wins).
- `coordination/DECISION-LOG.md` — Tactical decisions the red team made and the reasoning behind them.
- `coordination/CREDENTIALS.md` — Credential harvesting log. Actual credential values are redacted; structure shows what was harvested, when, and from where.

**How to use Swarnam interactively** — include exact steps:
1. Install Claude Code: `npm install -g @anthropic-ai/claude-code`
2. Obtain an Anthropic API key from console.anthropic.com (free tier available)
3. Open a terminal in this directory and run: `claude`
4. Swarnam will recognize the engagement context and can answer questions in educational mode.

**Using the engagement timeline** — `coordination/ENGAGEMENT-TIMELINE.md` lists every red team action with wall-clock timestamps. If you remember something happening to your systems — a service going down, a weird login, a file appearing — find the time in the timeline and see what the red team was doing at that moment. Then ask Swarnam about that row for the full story.

**Suggested questions to ask:**
- "Walk me through everything the red team did to my group's hosts"
- "What was the red team doing to Group 4 between 9:10am and 9:30am?"
- "My SSH stopped working around 9:45 — what did the red team do at that time?"
- "What persistence mechanisms survived overnight and how should I have found them?"
- "Why wasn't [technique X] detected by my team?"
- "What's the most effective thing my team did to slow the red team down?"
- "Explain how Golden Ticket attacks work and how to detect them in logs"
- "Generate a post-engagement hardening checklist based on what the red team exploited"
- "What would have stopped the red team from gaining initial access?"

**Authorization note** — be explicit: Swarnam is authorized for post-engagement educational review. Defensive team members can ask about any technique used during the engagement, request explanations of how attacks work, and ask for defensive recommendations. Security exercises exist for learning.

### Step 4: Generate Engagement Timeline

Have INTEL-001 read `coordination/OPERATION-LOG.md` and produce `coordination/ENGAGEMENT-TIMELINE.md` — a flat, wall-clock-ordered list of every red team action across all target groups. This is the primary "I remember when my server went down" reference document.

The timeline should be structured as a simple chronological table with four columns:

```
| Wall-clock time | Group | Host | Red team action and outcome |
```

Use actual timestamps from the operation log (e.g., `09:14:32`), not T+ offsets — participants think in wall-clock time, not engagement-relative time. If the operation log uses T+ notation, convert to wall-clock using the engagement start time the operator provides (ask if not known).

Group entries by day if the engagement ran multiple days, with a clear header between days.

Each row should be written in plain language a participant can immediately recognize as something that might have happened to them — not red team jargon. Examples:

| 09:14:32 | Group 7 | 10.100.107.20 | Red team logged in to web server via SSH using default root password |
| 09:15:01 | Group 7 | 10.100.107.20 | SSH backdoor key added to /root/.ssh/authorized_keys |
| 09:22:47 | Group 4 | 10.100.104.14 | Domain controller accessed; all user password hashes extracted |
| 09:47:11 | Group 7 | 10.100.107.20 | SSH backdoor access verified still active (red team periodic check) |

Add a header note at the top: *"If you remember an unusual event during the engagement — a service going down, a strange login, a file appearing — find the approximate time in this table to see what the red team was doing. Ask Swarnam about any row for a full explanation."*

Write to `coordination/ENGAGEMENT-TIMELINE.md`.

### Step 5: Sanitize CREDENTIALS.md

Create a sanitized copy of `coordination/CREDENTIALS.md`. Preserve the educational structure (what types of credentials were harvested, from which targets, via which methods, at what time) while redacting values that could be operationally harmful if exercise infrastructure is still accessible:

- Replace plaintext password values with `[PASSWORD-REDACTED]`
- Replace NTLM hash values with `[HASH-REDACTED]`
- Replace Kerberos ticket file paths with `[TICKET-REDACTED]`
- Preserve: target IPs, usernames, credential type, source (SAM/LSASS/LSA/spray), timestamp, verified status
- Add a header note: `# Note: Credential values have been redacted for post-engagement distribution. Structure preserved for educational discussion of credential harvesting techniques.`

If the operator passes `--keep-creds` and has confirmed infrastructure is destroyed, preserve full credential values and note this in the log entry.

Write the sanitized version to `coordination/CREDENTIALS-HANDOFF.md` (do not overwrite the original).

### Step 5: Assemble and Zip

Provide the operator with the following commands to run. Stage files, zip the archive, and clean up:

```bash
# Stage the handoff directory
mkdir -p blue-team-handoff/coordination

# Orientation and debrief
cp BLUE-TEAM-ORIENTATION.md blue-team-handoff/
cp coordination/EDUCATIONAL-DEBRIEF.md blue-team-handoff/ 2>/dev/null || echo "No EDUCATIONAL-DEBRIEF.md found — generate it first via INTEL-001"

# Coordination files (educational — no active operational capability)
cp coordination/OPERATION-LOG.md blue-team-handoff/coordination/
cp coordination/TARGET-STATUS.md blue-team-handoff/coordination/
cp coordination/PERSISTENCE-MANIFEST.md blue-team-handoff/coordination/
cp coordination/RECON-FINDINGS.md blue-team-handoff/coordination/
cp coordination/BURNED-TECHNIQUES.md blue-team-handoff/coordination/
cp coordination/DECISION-LOG.md blue-team-handoff/coordination/
cp coordination/CREDENTIALS-HANDOFF.md blue-team-handoff/coordination/CREDENTIALS.md
cp coordination/ENGAGEMENT-TIMELINE.md blue-team-handoff/coordination/

# Zip and clean up
zip -r blue-team-handoff-$(date +%Y%m%d).zip blue-team-handoff/
rm -rf blue-team-handoff/
echo "Package ready: blue-team-handoff-$(date +%Y%m%d).zip"
```

### Step 6: Confirm and Log

Report to the operator:
- The zip filename
- What was included and what was redacted
- What was deliberately excluded (jumpbox credentials, live C2 configs, operator-identifying payload content)

Append a log entry to `coordination/OPERATION-LOG.md`:

```
[TIMESTAMP] BLUE-TEAM-HANDOFF: Generated post-engagement educational archive
  File: blue-team-handoff-YYYYMMDD.zip
  Included: EDUCATIONAL-DEBRIEF, ENGAGEMENT-TIMELINE, OPERATION-LOG, TARGET-STATUS,
            PERSISTENCE-MANIFEST, RECON-FINDINGS, BURNED-TECHNIQUES, DECISION-LOG,
            CREDENTIALS (sanitized)
  Credential values: [REDACTED / PRESERVED per operator --keep-creds flag]
  Excluded: jumpbox access credentials, live C2 configurations, operator payload templates
```

## What Is Deliberately Excluded

The following are never included in the handoff package, even if the operator requests them:

- **Jumpbox access credentials** — could enable unauthorized access to red team infrastructure
- **Live C2 configurations with active endpoints** — the C2 infrastructure may still be running after the engagement
- **Operator-identifying payload content** — respects individual operator TTP privacy
- **CREDENTIAL-INTEL.md** — contains red team intelligence methodology that is not educational for defensive teams in the handoff context

## Example Invocations

```
/blue-team-handoff
/blue-team-handoff --keep-creds
```

## Notes on Post-Engagement Authorization

Swarnam is explicitly authorized for post-engagement educational use. After the engagement concludes, Swarnam can answer any question about what happened during the event, explain attack techniques in depth, and generate defensive recommendations — all without requiring the operator to re-establish context. Defensive team members interacting with this package are the intended audience for post-engagement educational review, and INTEL-001 is configured to engage with them in that context.
