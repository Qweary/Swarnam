## Workflow

This command is the application step of the training improvement cycle. It takes a patch file generated by TRAIN-003 and walks the operator through applying each edit to the competition files.

### Step 1: Parse the Patch File

Read the specified patch file and extract all edit blocks. For each edit, parse the target file path, section identifier, edit type (REPLACE, INSERT-AFTER, INSERT-BEFORE), before text, after text, and rationale.

Validate that the patch file has the expected structure. If any edit block is malformed (missing before/after sections, missing target file path), report the issue and skip that edit rather than aborting the entire patch.

Count the total edits and report: "Patch contains N edits across M files."

### Step 2: Verify Target Files Exist

For each unique target file referenced in the patch, verify the file exists at the specified path. If a target file doesn't exist, report which edit blocks reference it and ask the operator whether to skip those edits or abort.

### Step 3: Present Edits for Review

Walk through each edit sequentially. For each edit, present:

The edit number and short description from the patch file.

The source finding (debrief item number, disposition category, and agent).

The target file and section.

A contextual diff showing the before text highlighted within its surrounding context (5 lines above and below in the target file) and the after text that will replace it. For INSERT-AFTER and INSERT-BEFORE edits, show the anchor text and where the new content will be inserted.

The rationale from the patch file.

After presenting each edit, ask the operator for one of four decisions:

APPLY: apply this edit as written. This is the default and most common action.

MODIFY: the operator wants to adjust the after text before applying. Present the after text for editing, then apply the modified version. Record the modification in the training log.

SKIP: skip this edit entirely. Record the skip reason in the training log. The edit remains in the patch file for reference but is not applied.

ABORT: stop processing the patch entirely. Edits already applied in this session remain applied (they're individual file changes). Unapplied edits are not touched.

### Step 4: Apply Each Confirmed Edit

For each APPLY or MODIFY decision, make the file change:

For REPLACE edits: find the exact "before" text in the target file and replace it with the "after" text. If the "before" text doesn't appear exactly as specified (perhaps the file was already modified by a previous patch), report the mismatch and ask the operator whether to attempt a fuzzy match, skip, or abort.

For INSERT-AFTER edits: find the anchor text and insert the new content immediately after it.

For INSERT-BEFORE edits: find the anchor text and insert the new content immediately before it.

After each successful edit, display a brief confirmation: "Edit #{N} applied to {file}."

### Step 5: Cross-Agent Validation

After all edits are applied (or skipped), run a quick validation pass:

Check that any coordination file template changes are reflected in the agents that reference those files. If the patch changed a column in TARGET-STATUS.md but didn't include corresponding edits to agents that read/write that column, warn the operator.

Check that the modified files still pass the structural validation tests from the test framework (Test 1.2 for agent definitions, Test 1.3 for command definitions, Test 2.2 for coordination file templates). If any test would fail, report the specific failure.

### Step 6: Commit Changes

After validation, prepare a git commit with the structured commit message from the patch file. Present the commit message to the operator for review:

```
training: apply patch-{YYYYMMDD}-{N} from training run #{X}

[one-line summary per applied edit]
[skipped edits noted as "skipped: reason"]

Addresses N findings from training debrief {date}.
See training/patches/patch-{YYYYMMDD}-{N}.md for full rationale.
```

The operator can modify the commit message before confirming. After confirmation, stage the modified files and commit.

If git is not initialized in the workspace, skip the commit step and inform the operator which files were modified so they can manage version control manually.

### Step 7: Update Training Log

Append a patch application entry to training/TRAINING-LOG.md:

```
### Patch Applied: {date and time}

Patch file: {patch path}
Source run: Training Run #{X}
Edits applied: {count}
Edits skipped: {count}
Edits modified: {count}
Files changed: {list of file paths}
Commit: {commit hash or "manual"}
```

### Step 8: Next Steps

After applying the patch, inform the operator:

If all edits were applied: the swarm is ready for the next training run. The applied changes should be validated by running the modified agents against the same scenarios that produced the findings.

If some edits were skipped: note which findings remain unaddressed and whether they should be revisited in a future patch.

If this patch completes a training cycle (all patches from a debrief applied): suggest running the structure validation tests (Test Suite Category 1) to confirm the swarm's structural integrity, and then starting the next training run to measure improvement.

## Usage Examples

Apply a patch from the most recent debrief:
```
/apply-training training/patches/patch-20260315-1.md
```

The command walks through each edit interactively. There's no "apply all" mode — every edit gets operator review. This is deliberate: blind patch application is how regressions happen.
